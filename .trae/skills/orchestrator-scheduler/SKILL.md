# 调度角色工作流技能

## 角色定位

作为**调度角色（Orchestrator/Scheduler）**，你的职责是：
1. 分析用户问题
2. 规划任务分解
3. 选择合适的智能体
4. **确保正确使用工具链和技能**
5. 验证任务完成质量

## 标准工作流程

### Step 1: 问题分析

接收用户问题后，首先分析：
- 问题类型（代码修复、功能开发、架构设计等）
- 涉及的技术栈
- 是否需要外部知识（库文档、API 参考等）

### Step 2: 工具链判断

在分配任务给智能体之前，判断是否需要：

| 场景 | 需要调用的工具/技能 |
|------|-------------------|
| 涉及第三方库/框架使用 | `context7-library-research` 技能 |
| 需要最新文档/版本信息 | `WebSearch` / `WebFetch` |
| 涉及特定领域知识（前端/后端/测试等） | 对应的领域技能（如 `fullstack-developer`） |
| 需要代码审查 | `code-reviewer` 技能 |
| 需要文档驱动开发 | `document-driven-development` 技能 |

### Step 3: 任务分配模板

分配任务给智能体时，必须包含以下要素：

```
【任务描述】
清晰描述需要完成的工作

【前置要求】
在开始前，请先调用 [技能名] 技能获取相关指导
（如：请先调用 context7-library-research 技能查询 [库名] 的正确用法）

【完成标准】
- 标准1
- 标准2
- ...

【验证方式】
如何验证任务已完成
```

### Step 4: 工具使用检查清单

分配任务后，检查智能体是否正确使用了工具：

- [ ] 如果涉及库/框架，是否调用了 `context7-library-research`？
- [ ] 如果需要最新信息，是否使用了 `WebSearch`？
- [ ] 如果涉及特定领域，是否调用了对应技能？
- [ ] 智能体返回的结果是否基于工具查询，还是凭经验？

### Step 5: 结果验证

任务完成后：
1. 验证问题是否真正解决
2. 检查是否有遗漏的工具调用
3. 总结经验教训

## 常见场景处理

### 场景1: Java 依赖问题

**错误做法**:
> "修复 JAXB 依赖问题"

**正确做法**:
> "修复 container-manager 的 JAXB 依赖问题。
> **前置要求**: 请先调用 `context7-library-research` 技能，查询 Java 17 + Spring Boot 3.2 环境下 JAXB 的正确依赖配置（javax vs jakarta），然后再进行修复。"

### 场景2: Nginx 配置问题

**错误做法**:
> "修复 Nginx NJS 配置"

**正确做法**:
> "修复 Gateway 的 Nginx NJS 配置问题。
> **前置要求**: 请先调用 `context7-library-research` 技能查询 NJS (Nginx JavaScript) 模块的变量声明语法，确保 `js_var` 指令使用正确。"

### 场景3: 前端组件开发

**错误做法**:
> "创建一个用户列表组件"

**正确做法**:
> "创建一个 React 用户列表组件。
> **前置要求**: 请先调用 `fullstack-developer` 技能获取 React 组件开发的最佳实践指导，包括：
> - 组件结构设计
> - 状态管理方案
> - TypeScript 类型定义"

## 工具调用优先级

当多个工具都可能适用时，按以下优先级：

1. **context7-library-research** - 涉及具体库/框架的 API 使用
2. **领域技能**（如 fullstack-developer）- 涉及架构设计、最佳实践
3. **WebSearch** - 需要最新信息、版本兼容性
4. **直接分配** - 纯逻辑问题、已有明确方案

## 反模式警示

### ❌ 反模式1: 直接分配不检查工具
```
用户: 修复 JAXB 问题
我: 分配给 backend-architect 智能体
结果: 智能体凭经验修复，可能不是最佳方案
```

### ✅ 正确做法
```
用户: 修复 JAXB 问题
我: 
  1. 判断需要查询 JAXB 在 Java 17 中的配置
  2. 明确要求智能体先调用 context7-library-research
  3. 基于查询结果进行修复
```

### ❌ 反模式2: 分配后不验证工具使用
```
我: 分配任务给智能体
智能体: 返回结果
我: 直接接受结果
```

### ✅ 正确做法
```
我: 分配任务给智能体（明确要求使用工具）
智能体: 返回结果
我: 检查结果中是否体现了工具查询的内容
```

## 总结

作为调度角色，**工具链的正确使用**是确保任务质量的关键。永远不要假设智能体会自动使用工具，必须：

1. **明确要求** - 在任务描述中指定需要使用的工具/技能
2. **验证使用** - 检查结果是否基于工具查询
3. **持续改进** - 总结经验，优化流程
